# 架构治理原则

> 文档类型：架构审计与治理方法论
> 首次生效：2026-02-26
> 适用范围：所有对本仓库代码的新增、修改、重构
> 定位：本文档不定义"系统是什么"（见 SYSTEM_ARCHITECTURE.md），而定义"如何判断一个实现是否在伤害系统"。

## 1. 核心立场

功能总是可以实现的。任何需求都可以通过某种方式跑通——编译通过、测试通过、用户能用。

但"能跑"不等于"正确"。**架构的破坏往往隐藏在功能正常工作的表象之下。** 一段代码可以在机械意义上完全正确（无竞态、有错误处理、接口对齐、测试通过），但在工程实践意义上是一个反模式——它为了实现当前功能，选择了一条伤害系统结构的路径。

这种伤害是表层发现不了的。它不会导致编译失败，不会让测试变红，甚至短期内不会引发任何可观测的问题。它只会在某个时刻——可能是几天后、几周后——以一种看似无关的故障形式爆发，而排查者往往无法将故障追溯到最初的那个实现决策。

**架构治理的本质不是追踪代码质量指标，而是识别"为了实现功能而破坏架构"的反模式。**

## 2. 审视维度

架构审视在两个维度上进行：**模块内部**和**模块之间**。两者关注的问题不同，但都指向同一个核心判断——实现方式是否是该问题的正确解法。

### 2.1 模块实现审视（单模块维度）

对每一个模块（包、文件、struct），问一个问题：

> **为了让功能跑起来，这个模块的实现方式是否偏离了该问题的最佳工程实践？**

"最佳实践"不是指理论上最优雅的设计，而是指：一个有经验的工程师面对同样的需求和约束，会自然采用的实现方式。如果实际实现偏离了这个直觉，那就是一个反模式的信号——即使功能完全正确。

审视时关注的不是"代码写得好不好"，而是"实现路径选得对不对"。同一个功能可能有多条实现路径，其中一些路径虽然更短、更快，但会对模块自身或系统其他部分造成隐性伤害。

### 2.2 模块组织审视（跨模块维度）

对模块之间的关系，问一个问题：

> **这个模块的边界画对了吗？它和其他模块之间的依赖关系是否反映了它们在系统中的真实角色？**

模块组织的反模式不是"代码放错了文件"这么简单。它的本质是：模块之间的依赖方向、职责分工、抽象层次违背了系统的实际结构。例如，一个"消息处理"模块去直接编排"任务调度"，一个"命令行入口"模块去直接操作"数据持久层"——即使功能跑得通，这些依赖关系本身就在说明边界画错了。

## 3. 反模式的本质特征

反模式不能穷举为一个固定清单。清单可以作为历史教训的记录和入门参考，但治理不能局限于"对照清单检查"。真正的反模式识别需要从更一般的层面思考：**这个实现决策是在服务系统，还是在透支系统？**

一个实现是反模式，通常具备以下特征的组合：

1. **局部合理**：在当前功能的范围内，实现方式完全说得通，甚至是"最短路径"。
2. **表面无害**：编译通过、测试通过、功能正常，没有立即可观测的负面效果。
3. **隐性破坏**：它对系统其他部分的假设、生命周期、边界、一致性造成了伤害，但这种伤害只在特定条件下才显现。
4. **替代方案存在**：存在一种同样可以实现功能、但不会造成上述伤害的实现路径，且这条路径对有经验的工程师来说是显而易见的。

**判断一个实现是否是反模式的核心方法：不是问"它能不能工作"，而是问"一个对系统全貌有认知的工程师会不会这么实现"。** 如果答案是否定的，那就需要追问为什么——是信息不足导致的合理选择，还是为了走捷径而忽略了已知的更好方案。

## 4. 历史案例（作为参考而非穷举）

以下案例记录本项目中已发现并修复的反模式。它们的价值不在于"以后避免同样的错误"，而在于展示反模式的一般特征——看起来合理、功能正常、但实现路径选错了。

**治理时应当跳出这些具体案例，从第 3 节的一般性特征出发进行独立判断。**

### 案例 A：TTL 缓存销毁有状态资源

- **需求**：gateway 的 project resolver 需要支持配置热加载。
- **实现**：对整个 Project 实例加 30s TTL 缓存。过期后销毁旧实例，重新 `OpenProjectByName()` 创建新实例。
- **表面效果**：功能正常，配置变更会在 30s 内生效，并发测试通过。
- **实际伤害**：Project 持有 DB 连接、channel service、所有运行时状态。每 30s 这些资源被静默销毁并重建，导致 in-flight 操作丢失、状态不连续。
- **正确路径**：Project 实例跟 daemon 同生命周期（ProjectRegistry），配置变更通过 reload 机制更新，不销毁运行时资源。
- **反模式特征**：重建整个复合对象来更新其中一个属性（配置）。这是"局部最短路径"，但伤害了资源生命周期的完整性。

### 案例 B：方法内现场创建服务实例

- **需求**：channel 的 action executor 需要调用 PM/worker/ticket 服务来执行业务操作。
- **实现**：在每个 execute 方法内部 `pmsvc.New(project, workersvc.New(project))`，现场创建服务实例。
- **表面效果**：功能正常，每次调用都能拿到可用的服务。
- **实际伤害**：channel 包被迫 import pm/ticket/worker（平级互调），每次创建新实例意味着无法共享状态或连接，且测试时无法替换依赖。
- **正确路径**：依赖注入——在构造时从上层接收已初始化的服务，或通过接口抽象解耦。
- **反模式特征**：为了"在这里能用"，绕过了正常的依赖传递路径。

### 案例 C：逐字段复制代替直接传递

- **需求**：app 层需要把 channel service 的处理结果返回给调用者。
- **实现**：定义一套 app 层的 DTO struct，然后在方法内逐字段（20+ 个字段）手动赋值。
- **表面效果**：app 层有自己的类型定义，看起来实现了层间隔离。
- **实际伤害**：80+ 行样板代码，channel 每加一个字段 app 就要同步改，隔离是虚假的——两套 struct 的字段完全一致。
- **正确路径**：直接暴露 channel 的结果类型（如果 app 本来就不需要转换），或定义真正有区别的 DTO（如果确实需要不同的对外表示）。
- **反模式特征**：引入了"形式上的抽象"但没有"实质上的隔离"，增加维护成本却不带来真正的解耦。

## 5. 使用方式

本文档的使用场景：

1. **代码审查时**：对每个变更，从第 2 节的两个维度检视——模块实现路径是否正确，模块组织关系是否合理。
2. **架构审计时**：逐模块走查，不依赖固定清单，而是以第 3 节的一般性特征为判断标准。
3. **新功能设计时**：在选择实现路径之前，问一下"这条路径是否是一个有经验的工程师面对当前系统状态会选择的方式"。
4. **发现新反模式时**：将其补充到第 4 节作为案例，但始终提醒自己——案例是有限的，判断力才是根本。
