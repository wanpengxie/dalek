# 架构治理原则

> 文档类型：架构审计与治理方法论
> 首次生效：2026-02-26
> 适用范围：所有对本仓库代码的新增、修改、重构
> 定位：本文档不定义"系统是什么"（见 SYSTEM_ARCHITECTURE.md），而定义"如何判断一个实现是否在伤害系统"。

## 1. 核心立场

功能总是可以实现的。任何需求都可以通过某种方式跑通——编译通过、测试通过、用户能用。

但"能跑"不等于"正确"。**架构的破坏往往隐藏在功能正常工作的表象之下。** 一段代码可以在机械意义上完全正确（无竞态、有错误处理、接口对齐、测试通过），但在工程实践意义上是一个反模式——它为了实现当前功能，选择了一条伤害系统结构的路径。

这种伤害是表层发现不了的。它不会导致编译失败，不会让测试变红，甚至短期内不会引发任何可观测的问题。它只会在某个时刻——可能是几天后、几周后——以一种看似无关的故障形式爆发，而排查者往往无法将故障追溯到最初的那个实现决策。

**架构治理的本质不是追踪代码质量指标，而是识别"为了实现功能而破坏架构"的反模式。**

## 2. 审视维度

架构审视在两个维度上进行：**模块内部**和**模块之间**。两者关注的问题不同，但都指向同一个核心判断——实现方式是否是该问题的正确解法。

### 2.1 模块实现审视（单模块维度）

对每一个模块（包、文件、struct），问一个问题：

> **为了让功能跑起来，这个模块的实现方式是否偏离了该问题的最佳工程实践？**

"最佳实践"不是指理论上最优雅的设计，而是指：一个有经验的工程师面对同样的需求和约束，会自然采用的实现方式。如果实际实现偏离了这个直觉，那就是一个反模式的信号——即使功能完全正确。

审视时关注的不是"代码写得好不好"，而是"实现路径选得对不对"。同一个功能可能有多条实现路径，其中一些路径虽然更短、更快，但会对模块自身或系统其他部分造成隐性伤害。

### 2.2 模块组织审视（跨模块维度）

对模块之间的关系，问一个问题：

> **这个模块的边界画对了吗？它和其他模块之间的依赖关系是否反映了它们在系统中的真实角色？**

模块组织的反模式不是"代码放错了文件"这么简单。它的本质是：模块之间的依赖方向、职责分工、抽象层次违背了系统的实际结构。例如，一个"消息处理"模块去直接编排"任务调度"，一个"命令行入口"模块去直接操作"数据持久层"——即使功能跑得通，这些依赖关系本身就在说明边界画错了。

## 3. 反模式的本质特征

反模式不能穷举为一个固定清单。清单可以作为历史教训的记录和入门参考，但治理不能局限于"对照清单检查"。真正的反模式识别需要从更一般的层面思考：**这个实现决策是在服务系统，还是在透支系统？**

一个实现是反模式，通常具备以下特征的组合：

1. **局部合理**：在当前功能的范围内，实现方式完全说得通，甚至是"最短路径"。
2. **表面无害**：编译通过、测试通过、功能正常，没有立即可观测的负面效果。
3. **隐性破坏**：它对系统其他部分的假设、生命周期、边界、一致性造成了伤害，但这种伤害只在特定条件下才显现。
4. **替代方案存在**：存在一种同样可以实现功能、但不会造成上述伤害的实现路径，且这条路径对有经验的工程师来说是显而易见的。

**判断一个实现是否是反模式的核心方法：不是问"它能不能工作"，而是问"一个对系统全貌有认知的工程师会不会这么实现"。** 如果答案是否定的，那就需要追问为什么——是信息不足导致的合理选择，还是为了走捷径而忽略了已知的更好方案。

## 4. 历史案例（作为参考而非穷举）

以下案例记录本项目中已发现并修复的反模式。它们的价值不在于"以后避免同样的错误"，而在于展示反模式的一般特征——看起来合理、功能正常、但实现路径选错了。

**治理时应当跳出这些具体案例，从第 3 节的一般性特征出发进行独立判断。**

### 案例 A：TTL 缓存销毁有状态资源

- **需求**：gateway 的 project resolver 需要支持配置热加载。
- **实现**：对整个 Project 实例加 30s TTL 缓存。过期后销毁旧实例，重新 `OpenProjectByName()` 创建新实例。
- **表面效果**：功能正常，配置变更会在 30s 内生效，并发测试通过。
- **实际伤害**：Project 持有 DB 连接、channel service、所有运行时状态。每 30s 这些资源被静默销毁并重建，导致 in-flight 操作丢失、状态不连续。
- **正确路径**：Project 实例跟 daemon 同生命周期（ProjectRegistry），配置变更通过 reload 机制更新，不销毁运行时资源。
- **反模式特征**：重建整个复合对象来更新其中一个属性（配置）。这是"局部最短路径"，但伤害了资源生命周期的完整性。

### 案例 B：方法内现场创建服务实例

- **需求**：channel 的 action executor 需要调用 PM/worker/ticket 服务来执行业务操作。
- **实现**：在每个 execute 方法内部 `pmsvc.New(project, workersvc.New(project))`，现场创建服务实例。
- **表面效果**：功能正常，每次调用都能拿到可用的服务。
- **实际伤害**：channel 包被迫 import pm/ticket/worker（平级互调），每次创建新实例意味着无法共享状态或连接，且测试时无法替换依赖。
- **正确路径**：依赖注入——在构造时从上层接收已初始化的服务，或通过接口抽象解耦。
- **反模式特征**：为了"在这里能用"，绕过了正常的依赖传递路径。

### 案例 C：逐字段复制代替直接传递

- **需求**：app 层需要把 channel service 的处理结果返回给调用者。
- **实现**：定义一套 app 层的 DTO struct，然后在方法内逐字段（20+ 个字段）手动赋值。
- **表面效果**：app 层有自己的类型定义，看起来实现了层间隔离。
- **实际伤害**：80+ 行样板代码，channel 每加一个字段 app 就要同步改，隔离是虚假的——两套 struct 的字段完全一致。
- **正确路径**：直接暴露 channel 的结果类型（如果 app 本来就不需要转换），或定义真正有区别的 DTO（如果确实需要不同的对外表示）。
- **反模式特征**：引入了"形式上的抽象"但没有"实质上的隔离"，增加维护成本却不带来真正的解耦。

### 案例 D：类型镜像泛滥——app 层为 service 层维护完全相同的 DTO

- **背景**：架构重构将业务逻辑从 app（facade）层下沉到 service 层，service 层定义了自己的输入输出类型。
- **实现**：app 层为每个 service 类型都创建了字段完全相同的镜像 struct（如 `app.TicketView` 镜像 `ticket.TicketView`），并通过逐字段手工赋值进行转换。
- **表面效果**：app 层有"自己的"类型定义，看起来实现了层间解耦。
- **实际伤害**：8+ 组镜像类型、85+ 个字段的手工拷贝、7 个纯转换函数。service 层每新增一个字段，需要同步修改 3 处（service 定义、app 镜像定义、转换函数），漏改不会编译报错，只会静默丢失字段。
- **正确路径**：如果两套 struct 字段完全一致，app 层应使用 type alias（`type TicketView = ticketsvc.TicketView`）。只有在 app 层确实需要不同的字段表示时（如权限过滤、格式转换），才需要独立的 DTO。
- **反模式特征**：这是案例 C 的系统性翻版——在整个 facade 层批量引入了"形式上的抽象"。AI agent 在重构时选择了"安全"的保留策略（保留旧类型以防万一），但这种安全感是虚假的，它只是把"改一处"变成了"改三处"。

### 案例 E：FSM 约束只覆盖正常路径，异常路径全部绕过

- **背景**：重构引入了 `internal/fsm/` 包，为 ticket/worker/dispatch/task 定义了状态转换表和 guard 函数。
- **实现**：正常路径（dispatch、archive、worker report）通过 guard 函数校验状态转换合法性。但异常路径（recovery 恢复、merge 归档、direct dispatch 回滚）直接用 GORM `Updates()` 写入 `workflow_status`，不经过任何 FSM 校验。
- **表面效果**：FSM 存在、状态表定义完整、正常路径被约束、测试通过。
- **实际伤害**：至少 5 条路径完全绕过 FSM。更危险的是，guard 函数（如 `CanReportPromoteTo`）末尾有 `return true` 兜底——即使正常路径也只是"看起来"被约束。后续开发者看到 FSM 存在，会假设所有状态转换都受保护，但实际上异常路径是裸奔的。
- **正确路径**：所有状态写入点必须经过 FSM 校验，或者对豁免路径进行显式标注和审计。
- **反模式特征**：引入约束机制但不完整执行，比没有约束更危险——它创造了虚假的安全感。

### 案例 F：文件搬移但依赖关系未重构

- **背景**：重构将 feishu（2087 行）、notebook（1150 行）、subagent（526 行）从 app 层下沉到 service 层。
- **实现**：代码物理位置从 `internal/app/` 移到了 `internal/services/xxx/`，但依赖关系保持原样——新 service 直接持有 `*core.Project`（一个包含 DB、Tmux、Git、Config 等 12+ 字段的重量级对象），而实际只使用其中 2-3 个字段。
- **表面效果**：代码在正确的包中，app 层文件大幅瘦身，看起来完成了分层。
- **实际伤害**：service 层对 `core.Project` 的强依赖使得独立测试需要构造完整环境（testutil.NewTestProject 需要 349 行）。更重要的是，feishu service 搬移后仍是 2195 行单文件，4 种完全不同的职责（HTTP handler、命令处理、卡片构建、HTTP 客户端）挤在一起，最大的函数仍是 616 行的超级闭包。"下沉"解决了分层问题，但没有解决职责分离问题。
- **正确路径**：搬移只是第一步。完整的重构应该是：搬移 → 定义最小化依赖接口 → 注入依赖 → 拆分职责 → 独立测试。
- **反模式特征**：把"搬移"当作"重构"。文件在正确的位置给人一种"已完成"的错觉，但内部结构问题原封不动地带到了新位置。

## 5. 熔岩流与船锚（Lava Flow & Boat Anchor）

这是与反模式并列的第二类治理问题。反模式的本质是"选错了实现路径"，而熔岩流与船锚的本质是"旧的路径没有清理干净"。两者的来源不同，识别方式也不同。

### 熔岩流（Lava Flow）

**来源**：火山喷发时熔岩到处流，冷却后变成没人敢碰、没人知道能不能删的化石。

熔岩流是大规模重构后的典型后遗症：旧的类型定义、废弃的 adapter、不再使用的兼容层——它们功能上已经被替代，但物理上还存在于代码库中。随着时间推移，这些"化石"积累得越来越多，没有人清楚哪些还在被真正调用，哪些是历史遗留。

**识别信号**：
- 同一概念在两个包中各有一套类型定义，且字段完全相同
- 新旧 import 路径并存，消费方有的用新包、有的用旧包
- 架构意图已经明确（如"X 包是类型权威来源"），但代码实际引用仍然分散
- 函数或常量在两个以上的包中有语义相同的定义

**为什么特别危险**：熔岩流不影响编译，也不影响当前功能。它的伤害是在后续开发中：新开发者看到新旧两条路径并存，不知道该模仿哪条，往往选择旧的（因为旧的有更多使用示例）。它把本来应该一次性的迁移成本，变成了永久性的认知税。

**治理标准**：重构后必须搜索旧路径的所有引用，确认它们要么已删除，要么已改为 type alias / 转发调用，要么有明确的保留理由并注释说明。"暂时保留以防万一"不是理由——这正是熔岩流的起点。

### 船锚（Boat Anchor）

**来源**：把一个没用的船锚留在船上，"以防万一以后用得到"。

船锚是熔岩流的直接制造者，也是 AI agent 做重构时最常见的错误：为了降低删错东西的风险，把旧的保留、新的并行，结果旧的永远不会被删——因为永远可以找到"它还在被引用"的理由（通常是因为熔岩流本身就在引用它）。

**识别信号**：
- 注释掉的代码块（不是调试用的临时注释，而是"暂时保留"的旧实现）
- 带有 `// Deprecated`、`// Legacy`、`// TODO: remove` 标注但长期未删除的函数
- 未使用的字段（struct 中存在但没有任何调用路径赋值或读取的字段）
- "以防万一"的兼容层（一个类型转换函数，当前调用方为零）

**为什么特别危险**：船锚不仅自身占据认知空间，更会主动产生引用——后续开发者在不确定时，会调用那个"看起来能用"的旧函数，而不是去找正确的新路径。一个船锚足以阻止整个迁移的收口。

**治理标准**：重构时的删除和新建应该同步进行，而非"先新建、之后再删"。"之后"在工程实践中约等于"永远不会"。如果删除有顾虑，先以 type alias 或转发调用的形式明确标注"此处是转发，权威实现在 X"，而不是保留两套独立实现。

---

熔岩流与船锚的识别不依赖功能测试——功能测试无法发现它们。识别方法是：**在重构完成后，主动搜索旧路径、旧类型名、旧函数名，统计实际调用方数量，并与架构预期对比。** 引用数为零而实现仍然存在，是最直接的信号。

## 6. 使用方式

本文档的使用场景：

1. **代码审查时**：从两类治理问题出发检视变更——实现路径是否正确（反模式，参考第 2-4 节），旧路径是否清理干净（熔岩流与船锚，参考第 5 节）。
2. **架构审计时**：逐模块走查，不依赖固定清单，而是以第 3 节的一般性特征为判断标准。
3. **新功能设计时**：在选择实现路径之前，问一下"这条路径是否是一个有经验的工程师面对当前系统状态会选择的方式"。
4. **重构审查时**：对每次重构，除了验证新实现的正确性，还必须验证旧实现的清理完整性——搜索旧的类型名、函数名、import 路径，确认它们要么已删除、要么已改为别名/转发、要么有明确的保留理由。特别关注：新旧路径并存是否会误导后续开发者。
5. **约束机制引入时**：引入新的约束（FSM、架构测试、lint 规则）后，必须验证所有入口路径都已接入约束。未覆盖的路径不是"将来再补"——它是一个活跃的漏洞。
6. **发现新反模式时**：将其补充到第 4 节作为案例，但始终提醒自己——案例是有限的，判断力才是根本。
