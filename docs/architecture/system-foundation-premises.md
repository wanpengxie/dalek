# Dalek Foundation Premises

> 本文只记录两件已经确定的前提：
>
> 1. 我们依赖的执行底层基本性质是什么
> 2. 我们系统的目标是什么
>
> 本文不展开 `worker loop`、`report`、`handoff`、`acceptance` 等具体机制。

## 1. 为什么先写这份文档

在继续讨论 `ticket`、`run`、`worker`、`worker loop` 之前，必须先把不会变的前提钉死。

如果这两个前提不清楚，后续所有抽象都会漂：

- 要么把 agent 误当成确定性程序
- 要么把 agent 误当成几乎不可依赖的弱组件
- 要么把系统目标缩成“把一个 loop 跑通”

这三种都会把 dalek 带偏。

## 2. 我们依赖的执行底层基本性质

### 2.1 agent 是概率推理机

dalek 依赖的执行底层，不是传统确定性程序，而是带工具能力的 agent。

给定：

- 上下文
- 工具
- 指令

agent 会以一定概率：

- 正确理解任务
- 选择合适动作
- 执行动作
- 根据反馈继续调整
- 产出结果

模型能力越强、上下文越充分、工具反馈越清晰，这个概率通常越高。  
但这个概率永远不是 `1`。

### 2.2 agent 在单次运行内具备真实语义执行能力

agent 不是只会生成一段文本。

在一次 run 内，它会真实地：

- 读取代码和上下文
- 调工具、跑命令、改文件
- 观察结果
- 基于结果继续推理和调整动作

因此，agent 在单次运行里是一个真实的语义执行者，而不是“prompt 转文本”的外壳。

### 2.3 agent 的单次运行天然有边界，尤其受 context 限制

即使是强 agent，单次运行也不是无限能力容器。

它至少天然受这些边界约束：

- context 容量有限
- 单次运行时长有限
- 单次工具链路复杂度有限
- 单次可稳定维持的局部状态有限

这意味着：

- 单次 raw agent run 适合解决局部、有限、可收口的问题
- 它不能被假定刚好可以完整吞下一个复杂 ticket
- 一旦 ticket 目标超出单次运行容量，系统就必须把 ticket 推进拆成多次有边界的单步推进

### 2.4 agent 的原生产物是 trajectory 和 effects

一次 run 的自然产物首先是：

- `trajectory`
  消息、工具调用、观察结果、过程轨迹
- `effects`
  文件改动、测试输出、命令结果、外部副作用

也就是说，底层先有执行轨迹和执行结果，后面才可能在其上抽象出 `report`、`handoff`、`done` 等更高层对象。

### 2.5 agent 可以高概率遵守约定，但不能被假定为必然遵守

例如，一个 agent 在 run 完成后，可能有很高概率会正确执行 `report`。  
这个概率可以是 `99%`，但不会是 `100%`。

这意味着：

- agent 不是确定性真相机
- agent 也不是弱智组件
- 它是高能力但非零失误率的概率执行器

### 2.6 agent 能吃外部反馈并修正后续执行

agent 不是静态地反复采样。

只要外部世界给出明确反馈，它就可以：

- 理解反馈
- 修正判断
- 调整工具使用
- 继续执行下一步

这意味着，概率 agent 不是只能“再试一次”，而是可以在反馈驱动下做多轮收敛。

### 2.7 直接后果

从上面的性质直接推出：

- agent 可以作为 dalek 的主要执行力量
- agent 的单次运行不能直接承担 ticket 的完整推进
- agent 自身不能直接承担系统连续性
- dalek 不能把长期可靠性建立在“agent 会持续正确运行”这个假设上
- dalek 必须围绕 ticket 建立 loop，把复杂 ticket 拆成多次有边界推进

一句话说：

**agent 足够强，可以承担单次推进；但系统不能把整张 ticket 的完整推进和持续真相压在单次 raw run 上。**

## 3. 我们系统的目标

### 3.1 dalek 要解决的不是 demo loop，而是 repo 演化问题

dalek 不是要做一个“能跑通一次任务”的 agent demo。  
dalek 要解决的问题是：

**如何让 repo 的状态演化，通过 ticket 被持续、稳定、可治理地推进。**

### 3.2 ticket 是系统原语，不是普通任务单

在 dalek 里，`ticket` 不是单纯的 issue 或待办项。

它承载的是：

- 目标
- 约束
- 过程历史
- 推进语义
- 验收边界

也就是说，repo 的变化不应该只是“有人改了代码”，而应该是：

**某个 ticket 被系统性推进了。**

### 3.3 PM agent 是系统内核和维护者

dalek 的内核和维护者是 PM agent。  
这一点不变。

PM agent 负责：

- 基于 ticket 组织执行
- 判断推进
- 处理阻塞
- 接收澄清
- 决定下一步

因此，系统真正要服务的对象，不是单个 worker，而是 PM 对 ticket 的持续治理能力。

### 3.4 系统真正要达成的能力

dalek 的目标，不是把 agent 包成一个更复杂的 loop。  
dalek 真正要达成的是：

- 让强 agent 成为可持续使用的执行力量，而不是一次性 demo
- 让 ticket 成为 repo 演化的基本操作单元
- 让多次执行可以被接续、被恢复、被治理、被审计
- 让 PM agent 能持续推进开发，而不是反复猜底层 run 到底发生了什么

### 3.5 这两个前提合起来意味着什么

把第 2 节和第 3 节放在一起，得到一个最重要的结论：

**dalek 要解决的问题，不是“如何让 agent 永不犯错”，而是“在 agent 是概率执行器的前提下，如何把 ticket 做成 repo 演化的稳定系统原语”。**

这才是后续所有抽象必须服务的中心问题。

## 4. 后续讨论的起点

在这两个前提之上，后面才有资格继续讨论：

- `ticket execution` 和 `single step` 的边界分别是什么
- `worker` 和 `worker loop` 分别承担什么角色
- `report`、`handoff`、`acceptance` 这些机制分别属于哪一层
- dalek 真正值得下注的 foundation 到底放在哪
